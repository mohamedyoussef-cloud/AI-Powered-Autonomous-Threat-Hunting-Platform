# 1. Project Overview
## 1.1 Brief Description
The project aims to design and implement an AI-powered autonomous threat-hunting platform that supports Security Operations Center analysts in identifying, investigating, and reporting potential cyber threats.
The platform will use MITRE ATT&CK as its behavioral knowledge foundation. It will evaluate the organization’s operating environment, available telemetry, critical assets, previous incidents, and existing detection coverage to identify relevant threat-hunting opportunities.
Selected MITRE ATT&CK techniques will be transformed into explainable hunt hypotheses. A locally hosted large language model, such as Mistral, will then convert each approved hypothesis into a structured detection plan and a candidate Sigma detection rule.
Before execution, generated Sigma rules will pass through validation, field-availability checks, ATT&CK mapping verification, and query compilation. Valid detections will then be executed against supported security platforms, and the results will be correlated into structured hunt findings.
A separate machine-learning component will assist with threat scoring and false-positive suppression.
The proposed workflow is:
# 1.	Environment profiling
# 2.	MITRE ATT&CK technique filtering
# 3.	Technique scoring and ranking
# 4.	Explainable hunt-hypothesis generation
# 5.	AI-assisted detection synthesis
# 6.	Sigma rule validation
# 7.	Query compilation and execution
# 8.	Hunt finding generation
# 9.	Machine-learning-based triage
# 10.	Case management and reporting
The selected datasets and frameworks include MITRE ATT&CK Enterprise, the Sigma Rules Repository, Splunk BOTS v3, and EVTX-ATTACK-SAMPLES. DARPA Transparent Computing is retained as a possible future extension and is excluded from the core implementation because it would require a separate provenance-graph analysis architecture.
## 1.2 Business Context
Security Operations Centers commonly rely on alerts generated by existing security controls. This creates a largely reactive operating model in which analysts investigate events after a detection rule or security product has already identified suspicious activity.
Threat hunting provides a more proactive approach by allowing analysts to search for attacker behavior that may not have triggered an existing alert. However, effective threat hunting requires:
- Experienced security analysts
- Knowledge of MITRE ATT&CK
- Understanding of available log sources
- Familiarity with multiple query languages
- Time to design, execute, and document hunts
- Continuous maintenance of detection content
These requirements make threat hunting expensive, difficult to scale, and highly dependent on individual analyst experience.
The proposed platform addresses this context by automating selected parts of the threat-hunting lifecycle while preserving analyst visibility, validation controls, and explain ability.
## 1.3 Background
Traditional threat-hunting activities are often performed manually. Analysts select a threat behavior, determine whether the required telemetry is available, create platform-specific queries, review the results, correlate evidence, and prepare a report.
This process can become inconsistent across analysts and platforms. The same threat behavior may require separate queries for Splunk, Microsoft Sentinel, or Elastic.
The project addresses this issue by using Sigma as a vendor-neutral detection language. Instead of generating a different detection for every security platform, the system will generate a canonical Sigma rule and convert it into the appropriate query language through supported and validated conversion pipelines.
The project also separates system responsibilities:
- Deterministic logic will control technique eligibility and prioritization.
- The large language model will generate detection plans and candidate Sigma rules.
- The machine-learning model will evaluate hunt findings and assist with false-positive suppression.
- The security analyst will retain oversight of validation, execution, and final case decisions.

# 2. Business Problem and Opportunity
## 2.1 Business Problem
Organizations face several challenges when conducting proactive threat hunting:
# 1.	Threat hunting requires significant analyst time and expertise.
# 2.	Hunt selection may not be aligned with the organization’s actual systems, assets, or telemetry.
# 3.	Analysts may create detections that cannot be executed because required data sources or fields are unavailable.
# 4.	Detection logic must often be rewritten for different SIEM platforms.
# 5.	Manual query development may introduce syntax, field-mapping, or logical errors.
# 6.	Hunt findings may contain large numbers of benign events and false positives.
# 7.	Threat-hunting decisions are not always consistently documented or explainable.
# 8.	Existing detection coverage gaps may not be systematically identified.
# 9.	Hunt reports and case records may be created manually and inconsistently.
# 10.	Smaller or less mature security teams may lack dedicated threat-hunting specialists.
## 2.2 Business Opportunity
The project creates an opportunity to establish a repeatable, explainable, and partially autonomous threat-hunting process.
The platform can help organizations:
- Reduce the effort required to design threat hunts.
- Increase the number of relevant hunts performed.
- Align hunt activities with the organization’s actual risk profile.
- Identify telemetry and detection-coverage gaps.
- Standardize detection content through Sigma.
- Improve detection portability across security platforms.
- Reduce invalid or non-executable detection rules.
- Improve consistency in case creation and reporting.
- Assist less experienced analysts with ATT&CK-guided investigations.
- Retain human oversight while automating repetitive tasks.
The platform may also serve as a research and educational environment for evaluating the use of generative AI in security detection engineering.

# 3. Business Objectives
## 3.1 Measurable Goals
The following targets are proposed and should be validated during project initiation:
### Objective 1: Automate Hunt-Hypothesis Generation
Generate structured and explainable hunt hypotheses from prioritized MITRE ATT&CK techniques.
### Proposed measure
- At least 90% of generated hypotheses contain:
  - An ATT&CK technique identifier
  - A clear threat statements
  - Required data sources
  - Relevant assets or platforms
  - Supporting environmental evidence
  - A reason for prioritization
### Objective 2: Improve Detection Rule Validity
Produce candidate Sigma rules that comply with required syntax and schema standards.
### Proposed measures
- At least 85% Sigma YAML validation success
- At least 80% Sigma schema validation success
- At least 75% successful compilation for the selected target platform after refinement
### Objective 3: Prevent Unsupported Hunts
Ensure the platform does not automatically execute detections that require unavailable telemetry.
### Proposed measure
- 100% of proposed hunts must pass telemetry and field-availability checks before execution.
- Unsupported hunts must be classified as collection gaps.
### Objective 4: Demonstrate End-to-End Hunt Execution
Execute validated threat hunts against the selected evaluation datasets.
### Proposed measure
- Demonstrate at least three complete hunt campaigns covering different ATT&CK techniques or telemetry categories.
### Objective 5: Improve Finding Prioritization
Use machine learning to rank hunt findings and reduce likely false positives.
### Proposed measures
- Produce a threat probability score for each eligible hunt finding.
- Compare the machine-learning classifier against a defined baseline.
- Measure precision, recall, F1 score, and false-positive reduction where sufficient labels are available.
### Objective 6: Maintain Explain ability
Provide a traceable explanation for prioritization and rule-generation decisions.
### Proposed measure
- Every generated hunt must retain:
  - Technique scoring criteria
  - Environmental evidence
  - Required telemetry
  - Detection logic summary
  - Validation results
  - Execution status
### Objective 7: Produce Structured Reports
Generate an editable hunt report from each completed campaign.
### Proposed measure
- 100% of completed hunt cases should support report generation containing the hypothesis, evidence, findings, ATT&CK mapping, severity, analyst decision, and recommended actions.
## 3.2 Expected Business Outcomes
Expected outcomes include:
- Faster threat-hunt preparation
- More consistent selection of hunt targets
- Increased visibility into detection-coverage gaps
- Reduced generation of unusable queries
- Standardized detection content
- Improved analyst productivity
- Better documentation and auditability
- More efficient prioritization of suspicious findings
- Demonstration of a controlled application of generative AI in cybersecurity
- A reusable architecture for future threat-hunting capabilities

# 6. Business Requirements
## 6.1 High-Level Functional Requirements
### BR-FR-01: Environment Profile Management
The system shall allow authorized users to create, view, and update an environment profile.
The profile shall include supported platforms, telemetry sources, critical assets, incident history, and detection coverage.
### BR-FR-02: MITRE ATT&CK Data Ingestion
The system shall ingest and maintain relevant MITRE ATT&CK Enterprise data.
### BR-FR-03: Technique Eligibility Filtering
The system shall determine whether an ATT&CK technique is applicable to the profiled environment.
### BR-FR-04: Telemetry Readiness Validation
The system shall verify that required telemetry and fields are available before a hunt is approved for execution.
### BR-FR-05: Collection-Gap Identification
The system shall classify techniques with missing telemetry as collection gaps.
### BR-FR-06: Technique Prioritization
The system shall calculate a prioritization score for eligible ATT&CK techniques.
### BR-FR-07: Explainable Prioritization
The system shall display the criteria, scores, evidence, and reasoning used to prioritize each technique.
### BR-FR-08: Hunt-Hypothesis Generation
The system shall generate a structured hunt hypothesis from the selected technique and environmental context.
### BR-FR-09: Detection Plan Generation
The system shall use an LLM to generate a structured detection plan from an approved hunt hypothesis.
### BR-FR-10: Sigma Rule Generation
The system shall generate a candidate Sigma rule representing the proposed detection logic.
### BR-FR-11: Sigma Rule Validation
The system shall validate candidate Sigma rules before execution.
### BR-FR-12: ATT&CK Mapping Verification
The system shall verify that the generated rule contains the appropriate ATT&CK technique mapping.
### BR-FR-13: Field Validation
The system shall compare fields referenced by the generated rule against fields available in the selected telemetry source.
### BR-FR-14: Query Compilation
The system shall compile validated Sigma rules into a supported target query language.
### BR-FR-15: Query Execution
The system shall execute approved queries against an authorized data source.
### BR-FR-16: Execution Tracking
The system shall record query execution status, timestamps, errors, and returned result counts.
### BR-FR-17: Finding Generation
The system shall transform relevant query results into structured hunt findings.
### BR-FR-18: Evidence Correlation
The system shall associate related events and evidence with a common hunt case.
### BR-FR-19: Finding Scoring
The system shall assign a threat probability or prioritization score to eligible findings.
### BR-FR-20: False-Positive Suppression
The system shall identify findings that are likely to be false positives without permanently deleting the underlying evidence.
### BR-FR-21: Analyst Review
The system shall allow analysts to confirm, reject, edit, or escalate findings.
### BR-FR-22: Case Management
The system shall allow users to create and manage hunt cases.
### BR-FR-23: Reporting
The system shall generate an editable hunt report.
### BR-FR-24: Audit History
The system shall maintain a record of important decisions, validation results, executions, and analyst actions.
### BR-FR-25: Dashboard
The system shall provide a dashboard showing technique priorities, active hunts, validation status, findings, and case status.
### BR-FR-26: Dataset Evaluation
The system shall support rule testing and platform evaluation using BOTS v3 and EVTX-ATTACK-SAMPLES.
### BR-FR-27: Model Evaluation
The system shall compare the selected LLM generation approach against a baseline using defined quality metrics.
### BR-FR-28: Export
The system shall allow authorized users to export Sigma rules, query results, findings, and reports.
## 6.2 High-Level Non-Functional Requirements
### BR-NFR-01: Security
The platform shall protect sensitive configuration, telemetry, findings, and user information.
### BR-NFR-02: Access Control
The platform shall restrict functions according to assigned user roles.
### BR-NFR-03: Explain ability
The platform shall provide traceable reasoning for technique prioritization and hunt generation.
### BR-NFR-04: Reliability
The platform shall prevent invalid or unvalidated rules from being executed.
### BR-NFR-05: Accuracy
The platform shall measure rule validity, compilation success, field accuracy, and detection performance.
### BR-NFR-06: Performance
The platform should return interactive dashboard operations within an acceptable response time under the expected project workload.
A detailed performance threshold should be defined after prototype benchmarking.
### BR-NFR-07: Maintainability
The platform shall use modular components so that ATT&CK ingestion, hypothesis generation, validation, query execution, triage, and reporting can be maintained independently.
### BR-NFR-08: Portability
Detection logic shall be stored in a vendor-neutral Sigma format wherever possible.
### BR-NFR-09: Auditability
The platform shall retain sufficient records to reconstruct how a hunt was selected, generated, validated, executed, and reviewed.
### BR-NFR-10: Usability
The user interface shall present technical information in a form understandable to the intended analyst audience.
### BR-NFR-11: Data Integrity
The system shall preserve original evidence and distinguish it from generated summaries or model outputs.
### BR-NFR-12: Privacy
The system shall avoid transmitting sensitive telemetry to external AI services unless explicitly approved.
### BR-NFR-13: Local AI Operation
The platform should support local execution of the selected language model.
### BR-NFR-14: Scalability
The architecture should allow additional techniques, telemetry sources, and query backends to be added in future versions.
### BR-NFR-15: Error Handling
The system shall provide clear error messages for failed validation, compilation, connection, and execution operations.
### BR-NFR-16: Reproducibility
Evaluation experiments shall record the dataset version, model version, prompts, configurations, and metrics used.

# 8. Business Process
## 8.1 Current Process: As-Is
The expected current threat-hunting process is largely manual:
# 1.	An analyst reviews threat intelligence, alerts, incidents, or ATT&CK techniques.
# 2.	The analyst selects a behavior to investigate.
# 3.	The analyst manually checks whether relevant telemetry exists.
# 4.	The analyst identifies the required log source and fields.
# 5.	The analyst writes a platform-specific query.
# 6.	The analyst tests and corrects the query.
# 7.	The analyst executes the query.
# 8.	The analyst manually reviews a potentially large result set.
# 9.	The analyst correlates related events.
# 10.	The analyst determines whether the activity is suspicious.
# 11.	The analyst creates a case or investigation record.
# 12.	The analyst prepares a report.
# 13.	Detection or collection gaps may be documented separately or not documented consistently.
### As-Is Process Limitations
- High dependency on analyst experience
- Long preparation time
- Inconsistent hunt selection
- Repeated query-development effort
- Limited portability between SIEM products
- Risk of using unavailable fields
- Inconsistent documentation
- High false-positive review workload
- Limited visibility into detection-coverage gaps
- Difficulties reproducing previous hunts
## 8.2 Future Process: To-Be
The proposed future process is:
# 1.	An authorized user creates or updates the environment profile.
# 2.	The system imports current MITRE ATT&CK information.
# 3.	The system filters techniques according to platform and telemetry compatibility.
# 4.	Unsupported techniques are recorded as collection gaps.
# 5.	Eligible techniques are scored and ranked.
# 6.	The analyst reviews the ranking and supporting explanation.
# 7.	The analyst selects or approves a technique.
# 8.	The system creates a structured hunt hypothesis.
# 9.	The LLM generates a detection plan and candidate Sigma rule.
# 10.	The system validates the generated rule.
# 11.	Failed rules are returned for correction and are not executed.
# 12.	Validated rules are compiled into the target query language.
# 13.	The analyst or authorized workflow approves execution.
# 14.	The Query Runner executes the detection.
# 15.	Results are transformed into structured findings.
# 16.	Related evidence is correlated.
# 17.	The machine-learning component assigns threat probability and false-positive likelihood.
# 18.	The analyst reviews the prioritized findings.
# 19.	Confirmed findings are added to a hunt case.
# 20.	The platform generates an editable report.
# 21.	Detection coverage and collection-gap information are updated.
# 22.	Results are retained for audit and future model evaluation.
